# End-to-End PGO & Trace Pipeline Documentation

Этот документ предоставляет полное описание архитектуры, компонентов и процесса использования пайплайна для сбора Profile-Guided Optimization (PGO) статистики, извлечения CFG, наложения динамических трасс DynamoRIO и визуализации.

## 1. Введение

Для задач машинного обучения на компиляторах (например, RL-агентов) необходимы:
1. **Среда (Грамматика)** — Control Flow Graph (CFG), извлеченный из LLVM на самом последнем этапе перед генерацией машинного кода (Machine IR).
2. **PGO Статистика** — Вероятности переходов на основе профилирования реального выполнения, чтобы агент понимал "горячие" пути (hot paths).
3. **Ground Truth данные** — Реальные трассы выполнения (`traces`), которые на 100% совпадают со статическим графом (CFG), для обучения и валидации агента.

---

## 2. Компоненты системы

### 2.1 CFGDumper (LLVM Plugin)
`CFGDumper.so` — это плагин для LLVM, который работает на финальном этапе компиляции. Он извлекает граф потока управления в формате JSON.
- **LTO (Link-Time Optimization)**: Используется для получения Whole-Program CFG.
- **`.llvm_bb_addr_map`**: Специальная секция в бинарном файле, содержащая маппинг адресов базовых блоков на их внутренние `ID`. Это критически важно для связывания динамического трейса со статическим графом.
- **Извлекаемые метрики**: Количество инструкций (`instr_count`), вероятности переходов (`prob`), наличие вызовов (`has_call`, `call_target`), является ли блок точкой входа (`is_entry`).

### 2.2 InstrTracer (DynamoRIO Client)
`InstrTracer.so` — клиент DynamoRIO, который логирует выполнение программы.
- Из-за разницы в определении базовых блоков между LLVM и DynamoRIO, плагин логирует **каждую отдельную инструкцию**.
- Логируются не абсолютные адреса, а **Относительные смещения (RVA)** от начала загрузки модуля в памяти.
- Для высокой производительности используется кольцевой буфер `trace_buffer` размером 64KB.

---

## 3. Архитектура наложения трейсов (Trace Mapping)

Основная проблема заключалась в том, что DynamoRIO и LLVM по-разному определяют границы базовых блоков. Архитектура решения:
1. `InstrTracer` записывает сырые RVA каждой выполненной инструкции в `trace.bin`.
2. Из бинарника извлекается `.llvm_bb_addr_map` (маппинг LLVM) в файл `_bb_map.txt`.
3. Скрипт `tools_py/trace_pipeline.py` читает RVA из `trace.bin`, бинарным поиском находит соответствующий блок в `_bb_map.txt` и преобразует RVA в кортеж `(Имя функции, BB_ID)`.
4. Последовательность сжимается (удаляются дубликаты BB_ID) и **валидируется** (каждый переход проверяется на существование в графе `main.cfg.json`).
5. Идеальный Ground Truth трейс сохраняется в `compressed_trace.json`.

---

## 4. Пайплайн выполнения (6 этапов)

Пайплайн интегрирован в скрипт `scripts/full_pipeline.sh` и запускается через `Makefile`.

1. **Profile Generation Build**: Исходный код компилируется с флагами `-fprofile-instr-generate` и `-fcoverage-mapping` для создания бинарника, собирающего статистику.
2. **Profile Collection & Merge**: Инструментированный бинарник запускается, образуя `default.profraw`, который затем мержится в `code.profdata`.
3. **CFG Generation Build (with PGO)**: LTO-компиляция с передачей `-fprofile-instr-use=code.profdata`. Плагин `CFGDumper` извлекает CFG в `cfg.json`, где в поле `prob` лежат реальные вероятности PGO. Генерируется `.llvm_bb_addr_map`.
4. **DynamoRIO Trace Collection**: Бинарник запускается под управлением DynamoRIO с `InstrTracer`, генерируя `trace.bin`.
5. **Trace Compression & Validation**: `trace_pipeline.py` преобразует сырой RVA-трейс в сжатую последовательность BB_ID, валидирует её и сохраняет в `compressed_trace.json`.
6. **Visualization**: Скрипт `tools_py/visualize_cfg.py` генерирует SVG-изображения графов с подсветкой PGO и наложением трейсов.

---

## 5. Визуализация

Скрипт визуализации (`visualize_cfg.py`) использует `graphviz` и `cxxfilt` (для деманглинга C++ имён).
- **PGO Граф (`*_main_cfg_pgo.svg`)**: 
  - Ребра раскрашены в зависимости от вероятности: Красный (`prob >= 0.8`), Синий (`0.2 <= prob < 0.8`), Черный (`prob < 0.2`).
  - Стартовый и конечный узлы выделены рамками зеленого и фиолетового цветов соответственно.
  - Внешние вызовы (`Call`) выделены темно-красным жирным шрифтом.
- **Trace Граф (`*_main_cfg_pgo_trace.svg`)**: 
  - На граф накладывается `compressed_trace.json`. 
  - Посещенные блоки закрашиваются градиентом зеленого (в зависимости от числа посещений `Hits`). 
  - Посещенные ребра выделяются зеленым цветом.

## 6. Ограничения и Известные Особенности

- **Межпроцедурные переходы (Call/Return)**: Скрипт валидации отмечает переходы между разными функциями как "Межпроцедурные" и не требует их наличия как явных ребер внутри функционального JSON-графа (так как CFG в `main.cfg.json` разбит по функциям).
- **Исчерпывающий (Global) Trace**: В файл `compressed_trace.json` пишется единый, плоский, исчерпывающий массив базовых блоков в том порядке, в котором по ним шел процессор. Этот массив включает проваливания внутрь функций и возвраты из них, обеспечивая идеальный "Ground Truth" лабиринт выполнения для всей программы.
- **Рекурсивные вызовы**: Пайплайн валидации корректно распознает и обрабатывает рекурсивные вызовы (переход из блока с вызовом в начальный блок этой же функции) и возвраты из рекурсии (переход из финального блока обратно).
- **Внешние библиотеки / Исключения**: Если программа перепрыгивает в неинструментированную `libc` (или выбрасывает C++ исключение) и возвращается, трейс не прерывается. DynamoRIO логирует последнюю инструкцию блока с вызовом и первую инструкцию блока возврата.
- **Точность PGO**: PGO статистика собирается только для тех путей, которые были выполнены во время профилирования.
