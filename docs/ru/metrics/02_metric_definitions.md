Этот файл — вторая часть материала «трассы и метрики»; см. также [01_trace_levels_and_context.md](01_trace_levels_and_context.md) и [03_cli_scenarios_appendices.md](03_cli_scenarios_appendices.md).

## 4. Математические метрики сравнения (реализация в `trace_synthesizer.metrics`)

Ниже даны определения в том виде, в каком они реализованы в коде. Параметры по умолчанию можно переопределить через `MetricContext` и CLI.

### 4.1. Эмпирическое распределение по конечному алфавиту и сглаживание

Пусть задан конечный алфавит символов \(S\) (например, множество идентификаторов базовых блоков или множество ориентированных рёбер графа). По конечной выборке (пул посещений или пул переходов из одной или нескольких трасс) строятся счётчики \(c(s)\).

Для сравнения двух эмпирических распределений на **объединённой поддержке** \(S' = \{s: c_{\mathrm{ref}}(s) > 0 \vee c_{\mathrm{cand}}(s) > 0\}\) используется сглаженная оценка вероятности

\[
\hat p(s) = \frac{c(s) + \varepsilon}{\sum_{t \in S'} (c(t) + \varepsilon)},
\]

где \(\varepsilon > 0\) — малый константный параметр (по умолчанию `1e-8`), исключающий нулевые вероятности в знаменателе при вычислении дивергенции.

### 4.2. Метрика `block_visit_kl` (KL по частотам посещения блоков)

Для корпуса интра-трасс одной функции подсчитывается число вхождений каждого `bb` (с учётом всех посещений во всех трассах корпуса). Получаются две оценки \(\hat P\) (эталон) и \(\hat Q\) (кандидат).

**Основное значение метрики** (поле `value`): дивергенция Кульбака–Лейблера

\[
D_{\mathrm{KL}}(\hat P \,\|\, \hat Q) = \sum_{s \in S'} \hat P(s) \log \frac{\hat P(s)}{\hat Q(s)}.
\]

В структуре `details` дополнительно выводится симметризованная величина \(\frac{1}{2}(D_{\mathrm{KL}}(\hat P\|\hat Q) + D_{\mathrm{KL}}(\hat Q\|\hat P))\) и размер поддержки.

**Интерпретация.** Нулевое или малое значение означает близость *маргинального* распределения времени, проведённого в блоках. Это соответствует формулировке из proposal о сравнении распределений «посещений блоков» как упрощённого варианта сравнения распределений путей.

**Ограничение.** Маргиналь по блокам не различает перестановку порядка посещений с теми же частотами; для порядковых эффектов используются переходы и n-граммы (ниже).

### 4.3. Метрика `edge_transition_kl` (KL по парам соседних блоков)

Для каждой трассы извлекаются пары \((bb_i, bb_{i+1})\) вдоль последовательности (после фильтрации по функции). Подсчёт ведётся по всему корпусу. Далее строятся \(\hat P\) и \(\hat Q\) на алфавите рёбер наблюдаемого подграфа и вычисляется \(D_{\mathrm{KL}}(\hat P \,\|\, \hat Q)\).

**Интерпретация.** Это оценка близости **марковского приближения первого порядка** к реальным данным: насколько часто генератор совершает те же локальные переходы, что и эталон.

### 4.4. Метрика `hot_path_ngram_overlap` (пересечение частых n-грамм)

Для каждого \(n \in [n_{\min}, n_{\max}]\) (по умолчанию с 2 по 4) на корпусе эталонных трасс строится частотный ранг **n-грамм** идентификаторов блоков. Берётся множество \(T_{\mathrm{ref}}\) из \(K\) наиболее частых n-грамм (по умолчанию `top_k` = 64). Аналогично для корпуса кандидатов — \(T_{\mathrm{cand}}\).

**Показатель recall@K для эталона:** доля n-грамм из \(T_{\mathrm{ref}}\), которые встречаются хотя бы один раз в корпусе кандидата.

**Показатель Jaccard по «топам»:** \(|T_{\mathrm{ref}} \cap T_{\mathrm{cand}}| / |T_{\mathrm{ref}} \cup T_{\mathrm{cand}}|\).

**Агрегированное `value`:** среднее значение recall по всем использованным \(n\).

**Интерпретация.** Метрика приближает формулировку proposal о пересечении наиболее частых *последовательностей блоков* («горячие пути» в лексикографии n-грамм).

**Ограничение.** Выбор \(K\) и диапазона \(n\) влияет на чувствительность; для диплома рекомендуется фиксировать эти гиперпараметры в таблице эксперимента.

### 4.5. Метрика скорости (`metrics-bench-speed`)

Реализовано измерение **чистого времени CPU** на выполнение \(N\) эпизодов `rollout_episode` с `RandomPGOAgent` в текущем процессе Python. Это соответствует части proposal о сравнении времени генерации \(N\) синтетических трасс.

Время сбора \(N\) эквивалентных трасс DynamoRIO **не измеряется** автоматически: его необходимо получить внешним замером (скрипт пайплайна, журнал прогона) и передать в CLI как `--dynamo-seconds` (синоним `--dynamo-sec-for-1000` из формулировки proposal для \(N=1000\)) для той же величины \(N\). Тогда вычисляется отношение временей (ускорение) в функции `speedup_vs_dynamo`.
